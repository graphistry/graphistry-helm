# Troubleshooting, Debugging, and Verification Guide

This guide follows the deployment stages from the platform deployment guides ([k3s](k3s/README.md), [GKE](gke/README.md), [Tanzu](tanzu/README.md), [Cluster](cluster/README.md)). For each stage, it provides verification commands with expected sample output, and troubleshooting steps for common failures.

For platform-specific issues (DCGM on GKE/Tanzu, PSA on Tanzu, NSX networking), see the respective platform README.

### Troubleshoot and Debug

**"The connection to the server was refused":**

`kubectl` uses a kubeconfig file to know which cluster to connect to. This file contains the cluster API server address, authentication credentials, and the current context. If `kubectl` cannot find a valid kubeconfig, or if the cluster is not running, you will see "The connection to the server was refused".

Each Kubernetes distribution stores the kubeconfig in a different location:

| Distribution | Kubeconfig location | Documentation |
|---|---|---|
| k3s | `/etc/rancher/k3s/k3s.yaml` | [k3s docs](https://docs.k3s.io/cluster-access) |
| GKE | `~/.kube/config` (generated by `gcloud container clusters get-credentials`) | [GKE docs](https://cloud.google.com/kubernetes-engine/docs/how-to/cluster-access-for-kubectl) |
| Tanzu | `~/.kube/config` (generated by `tanzu cluster kubeconfig get`) | [Tanzu docs](https://techdocs.broadcom.com/us/en/vmware-tanzu/standalone-components/tanzu-kubernetes-grid/2-5/tkg/workload-clusters-examine.html) |

Check which kubeconfig `kubectl` is using:

```bash
# The KUBECONFIG env var takes precedence, then ~/.kube/config
echo $KUBECONFIG # e.g. output for k3s: /etc/rancher/k3s/k3s.yaml

# k3s sets up kubectl automatically, verify it works
kubectl get nodes
```

If `kubectl get nodes` works now, the cluster recovered on its own (e.g., after a machine reboot). If it still fails, verify the cluster service is running:

```bash
# k3s
sudo systemctl status k3s

# Check the API server is listening
curl -k https://127.0.0.1:6443/healthz
```

Example output for k3s (A `401 Unauthorized` response means the API server is running and reachable, the request was just not authenticated (expected for a raw curl without credentials). If you get `Connection refused` instead, the API server is not running. Restart it with `sudo systemctl restart k3s`):
```json
{
  "kind": "Status",
  "apiVersion": "v1",
  "metadata": {},
  "status": "Failure",
  "message": "Unauthorized",
  "reason": "Unauthorized",
  "code": 401

```

## 2. GPU Support

### Check Current GPU State

Before installing the GPU Operator, check what the cluster already knows about GPUs.

Check if the node reports GPU resources to Kubernetes:

```bash
kubectl get nodes -o go-template='{{range .items}}{{.metadata.name}}: nvidia.com/gpu={{index .status.capacity "nvidia.com/gpu"}}{{"\n"}}{{end}}'
```

Example output: `host12: nvidia.com/gpu=2`

- `nvidia.com/gpu=1` (or higher, which represents the number of attached GPUs, for instance `nvidia.com/gpu=2` for 2 GPUs): GPU is registered and available. The GPU Operator (or Device Plugin) is already working, skip to verification below.
- `nvidia.com/gpu=0`: The node sees the GPU hardware but no device plugin is registering it with Kubernetes. The GPU Operator needs to be installed.
- `nvidia.com/gpu=<no value>`: Kubernetes is not aware of any GPU. The NVIDIA driver may not be installed on the host.

Check if the NVIDIA driver is installed on the host:

```bash
nvidia-smi
```

If `nvidia-smi` works, the driver is installed and you should install the GPU Operator with `--set driver.enabled=false`. If it fails ("command not found" or "driver not loaded"), install the GPU Operator with `--set driver.enabled=true`.

Check if the GPU Operator is actually installed (not just leftover labels):

```bash
# Check for a helm release in the gpu-operator namespace
helm list -n gpu-operator

# Or check if the namespace and pods exist
kubectl get pods -n gpu-operator
```

If both commands return empty results, the GPU Operator is not installed. However, you may still see GPU-related output from `kubectl cluster-info dump | grep gpu`. There are two types of GPU entries in the node JSON, and it is important to distinguish them:

**Node labels** (set by the GPU Operator's GPU Feature Discovery, removable):
```bash
"nvidia.com/gpu.deploy.device-plugin": "true",
"nvidia.com/gpu.deploy.driver": "pre-installed",
"nvidia.com/gpu.present": "true",
...
```

**Node capacity fields** (set by the kubelet automatically, not removable via `kubectl label`):
```bash
"nvidia.com/gpu": "0",
```

To see only GPU-related node labels:

```bash
kubectl get node <node-name> -o go-template='{{range $k, $v := .metadata.labels}}{{$k}}={{$v}}{{"\n"}}{{end}}' | grep gpu
```

If you see `nvidia.com/gpu.deploy.*` or `nvidia.com/gpu.present` labels but the GPU Operator is not installed, these are stale labels from a previous GPU Operator installation. They do not mean the GPU Operator is running. To clean them up:

```bash
# List which GPU labels exist on the node first
kubectl get node <node-name> -o go-template='{{range $k, $v := .metadata.labels}}{{$k}}={{$v}}{{"\n"}}{{end}}' | grep gpu

# Remove them (ignore "not found" errors for labels that don't exist on your node)
kubectl label node <node-name> \
    nvidia.com/gpu-driver-upgrade-state- \
    nvidia.com/gpu.deploy.container-toolkit- \
    nvidia.com/gpu.deploy.dcgm- \
    nvidia.com/gpu.deploy.dcgm-exporter- \
    nvidia.com/gpu.deploy.device-plugin- \
    nvidia.com/gpu.deploy.driver- \
    nvidia.com/gpu.deploy.gpu-feature-discovery- \
    nvidia.com/gpu.deploy.node-status-exporter- \
    nvidia.com/gpu.deploy.operator-validator- \
    nvidia.com/gpu.present- \
    nvidia.com/gpu-driver-upgrade-enabled-
```

> **Note:** After cleanup, `kubectl cluster-info dump | grep gpu` will still show `"nvidia.com/gpu": "0"`. This is normal. That entry comes from the node's capacity fields (not labels) and is managed by the kubelet.

**Tanzu/vSphere users:** On Tanzu, GPUs are exposed to Kubernetes worker nodes through vSphere **VM Classes** configured with either PCI passthrough (DirectPath I/O) or vGPU profiles. If none of the commands above detect a GPU, the worker node VM was likely provisioned without a GPU-enabled VM class. This is a vSphere administrator task (creating the VM class, adding the PCI device, associating it with the vSphere Namespace). See the [Tanzu README](tanzu/README.md) for the full GPU passthrough vs vGPU setup, and consult your vSphere administrator. Additionally, the `gpu-operator` namespace on Tanzu requires a Pod Security Admission label before installing the GPU Operator:

```bash
kubectl label --overwrite ns gpu-operator pod-security.kubernetes.io/enforce=privileged
```

### Verification (after GPU Operator is installed)

After installing the GPU Operator (see your platform README for the exact command), verify the pods are running. The install command differs by platform:

```bash
# k3s (host driver already installed):
helm install --wait --generate-name \
    -n gpu-operator --create-namespace nvidia/gpu-operator \
    --set driver.enabled=false \
    --timeout 60m

# Tanzu/vSphere (GPU Operator installs the driver on the node):
helm install --wait --generate-name \
    -n gpu-operator nvidia/gpu-operator \
    --set driver.enabled=true \
    --set driver.version="<DRIVER_VERSION>" \
    --set cdi.enabled=true \
    --set cdi.default=true \
    --set 'toolkit.env[0].name=NVIDIA_RUNTIME_SET_AS_DEFAULT' \
    --set-string 'toolkit.env[0].value=true' \
    --timeout 60m
```

On Tanzu, the `gpu-operator` namespace must already exist with the privileged PSA label (see the Tanzu note in "Check Current GPU State" above). The `driver.version` must match your CUDA compatibility requirements (see the driver compatibility table below). See the [Tanzu README](tanzu/README.md) for the full list of flags and vGPU-specific configuration.

Check that all GPU Operator pods are running (use `--watch` to wait until they settle, press Ctrl+C when done):

```bash
kubectl get pods -n gpu-operator --watch

Check the NVIDIA container runtime is available:

```bash
kubectl get runtimeclass
```

Example output for k3s (look for `nvidia` in the list):

```bash
NAME                  HANDLER               AGE
crun                  crun                  30d
nvidia                nvidia                8d
nvidia-cdi            nvidia-cdi            54m
nvidia-experimental   nvidia-experimental   30d
nvidia-legacy         nvidia-legacy         54m
```

The `nvidia` runtime class is the one Graphistry uses. The other entries (crun, spin, wasmedge, etc.) are additional runtimes that k3s ships with and can be ignored. If `nvidia` is missing, the GPU Operator did not install the NVIDIA Container Toolkit correctly.

How the NVIDIA runtime becomes the default differs by platform:
- **k3s**: Set `--default-runtime nvidia` in the k3s systemd service (e.g., `ExecStart` in `/etc/systemd/system/k3s.service`).
- **Tanzu/vSphere**: Set `toolkit.env NVIDIA_RUNTIME_SET_AS_DEFAULT=true` in the GPU Operator helm install (already included in the Tanzu install command above). This configures containerd to use the nvidia runtime for all containers without needing explicit RuntimeClass references.

Verify nvidia-smi works from the GPU Operator's device plugin pod:

```bash
kubectl exec -n gpu-operator $(kubectl get pods -n gpu-operator -o name | grep device-plugin) \
    -- nvidia-smi
```

This confirms the driver is functional and shows the GPU model, driver version, CUDA version, memory, and temperature. If this command fails, the driver is not working correctly and must be fixed before proceeding. See [Troubleshoot and Debug](#troubleshoot-and-debug-1) below.

Verify the GPU is usable by Graphistry CUDA containers. The previous step confirmed the GPU Operator pods can access the GPU. This step verifies that a standalone CUDA container can also use it, which is how Graphistry services run. Graphistry v2.50.1+ supports CUDA 12 and CUDA 13 (see driver compatibility table below).

```bash

(CUDA_VERSION=12.8.0; GPU_TEST_POD=graphistry-gpu-test-$$; \
  kubectl delete pod $GPU_TEST_POD --ignore-not-found && \
  kubectl run $GPU_TEST_POD --restart=Never --image=nvidia/cuda:${CUDA_VERSION}-base-ubuntu22.04 -- nvidia-smi && \
  kubectl wait --for=jsonpath='{.status.phase}'=Succeeded pod/$GPU_TEST_POD --timeout=300s && \
  kubectl logs $GPU_TEST_POD; \
  kubectl delete pod $GPU_TEST_POD --ignore-not-found)
```

**Driver compatibility for Graphistry v2.50.1+ CUDA images** (sources: [RAPIDS Platform Support](https://docs.rapids.ai/platform-support/), [CUDA Toolkit Release Notes](https://docs.nvidia.com/cuda/cuda-toolkit-release-notes/)):

Graphistry v2.50.1+ uses RAPIDS 26.02 and publishes two flavors: CUDA 12 and CUDA 13. The `cuda.version` chart value accepts `"12"` or `"13"`. Internally, Graphistry builds on top of RAPIDS base images (`rapidsai/base:26.02-cuda12-py3.10` and `rapidsai/base:26.02-cuda13-py3.10`), which ship specific CUDA toolkit versions that determine the minimum driver requirement:

| Graphistry Build | RAPIDS | CUDA Toolkit in Image | Recommended Min Driver | Verified On |
|---|---|---|---|---|
| `cuda.version: "12"` | 26.02 | 12.9.1 | R575+ (575.51.03+) | k3s: R575 (575.57.08), R580 (580.126.20), R590 (590.44.01), R595 (595.58.03). GKE: R570 (570.133.20, T4, forward compat) |
| `cuda.version: "13"` | 26.02 | 13.1.0 | R590+ (590.44.01+) | k3s: R590 (590.44.01), R595 (595.58.03). Docker compose: R590 (590.48.01) |

We recommend the driver versions in the table above. Older drivers may work via NVIDIA's [forward compatibility](https://docs.nvidia.com/deploy/cuda-compatibility/) layer but are not verified by Graphistry.

The CUDA 13 flavor requires R590+ because the RAPIDS 26.02 base image (`rapidsai/base:26.02-cuda13-py3.10`) bakes CUDA 13.1 runtime (`CUDA_VERSION=13.1.0`), not 13.0. Drivers in the R580 range support CUDA 13.0 but not 13.1, so they are compatible with the CUDA 12 flavor only.

For example, driver 580.126.20 (reporting CUDA 13.0 in nvidia-smi) works with Graphistry's CUDA 12 build but fails with the CUDA 13 build because CUDA 13.1 requires R590+.

**NVIDIA reference documentation:**

- [CUDA Toolkit Release Notes](https://docs.nvidia.com/cuda/cuda-toolkit-release-notes/): Complete CUDA-to-driver version compatibility table for all CUDA releases.
- [CUDA Compatibility Guide](https://docs.nvidia.com/deploy/cuda-compatibility/): Explains minor version compatibility, forward compatibility, and the boundaries between CUDA major versions.
- [GPU Operator / Driver Component Matrix](https://docs.nvidia.com/datacenter/cloud-native/gpu-operator/latest/platform-support.html#gpu-operator-component-matrix): Supported Kubernetes versions (1.30-1.35), container runtimes (containerd, CRI-O), operating systems, cloud providers (EKS, GKE, AKS), and GPU architectures for each GPU Operator release. Check this before installing to confirm your platform is supported.
- [GPU Operator on GKE](https://docs.nvidia.com/datacenter/cloud-native/gpu-operator/latest/google-gke.html): GKE-specific instructions for the GPU Operator, including the `RUNTIME_CONFIG_SOURCE=file` workaround for GKE 1.33+ containerd config issue.

### Verification (NVIDIA Device Plugin without GPU Operator)

If you are using the NVIDIA Device Plugin directly (Option 2 in the platform READMEs) instead of the full GPU Operator, the device plugin only registers `nvidia.com/gpu` as a **resource** in the node's Capacity/Allocatable. It does **not** add the `nvidia.com/gpu.present=true` node **label**. That label is set by GPU Feature Discovery (GFD), which is a component of the GPU Operator.

Graphistry's default `nodeSelector` requires `nvidia.com/gpu.present: "true"`. Without this label, all Graphistry pods will be stuck in `Pending` with:

```
0/1 nodes are available: 1 node(s) didn't match Pod's node affinity/selector
```

After installing the device plugin, verify the GPU resource is registered:

```bash
kubectl get nodes -ojson | jq '.items[] | {name: .metadata.name, gpu: .status.capacity["nvidia.com/gpu"]}'
```

Expected output:

```json
{
  "name": "your-node",
  "gpu": "1"
}
```

Then manually add the required label on each GPU node:

```bash
kubectl label node <node-name> nvidia.com/gpu.present=true
```

Verify the label was applied:

```bash
kubectl get nodes --show-labels | grep "nvidia.com/gpu.present"
```

> **Note:** This manual label is not managed by any operator and will persist across reboots. If you add new GPU nodes later, remember to label them as well.

### Troubleshoot and Debug

**GPU not detected (`nvidia.com/gpu` shows 0 or `<no value>` after installing the operator):**

Work through these checks in order. The first one that fails is your root cause.

```bash
# 1. Are all GPU Operator pods running?
kubectl get pods -n gpu-operator

# 2. Is the device plugin running and healthy?
kubectl get pods -n gpu-operator | grep device-plugin
kubectl logs -n gpu-operator $(kubectl get pods -n gpu-operator -o name | grep device-plugin)

# 3. Check the GPU Operator controller logs for errors
kubectl logs -n gpu-operator deployment/gpu-operator

# 4. Is the node tainted, preventing operator DaemonSets from scheduling?
kubectl describe node <node-name> | grep -A5 Taints

# 5. Does the node have the GPU hardware at all? (PCI device check)
kubectl get nodes -o go-template='{{range .items}}{{.metadata.name}}{{"\n"}}{{range $k, $v := .metadata.labels}}{{if eq $k "feature.node.kubernetes.io/pci-10de.present"}}  {{$k}}={{$v}}{{"\n"}}{{end}}{{end}}{{end}}'
```

The label `feature.node.kubernetes.io/pci-10de.present=true` means Node Feature Discovery detected an NVIDIA PCI device (vendor ID `10de`). If this label is missing, the node has no NVIDIA GPU hardware visible to the OS.

**GPU Operator pods CrashLoopBackOff:**

First identify which pod is failing:

```bash
kubectl get pods -n gpu-operator | grep -E "Error|CrashLoop|Init"
kubectl describe pod -n gpu-operator <failing-pod-name>
kubectl logs -n gpu-operator <failing-pod-name>
```

Common causes by pod type:

| Failing Pod | Cause | Fix |
|---|---|---|
| `nvidia-driver-daemonset` | Driver already installed on host | Set `driver.enabled=false` in helm install |
| `nvidia-driver-daemonset` | No internet (driver container downloads OS packages) | Set up a local package mirror or use a pre-built driver image |
| `nvidia-driver-daemonset` | Kernel version mismatch with driver container | Update the GPU Operator version or use a compatible OS image |
| `nvidia-driver-daemonset` (Tanzu vGPU) | ESXi host vGPU Manager version incompatible with guest driver | Match `driver.version` to the host VIB version. Check with `esxcli software vib list \| grep nvidia` on the ESXi host |
| `nvidia-container-toolkit` | containerd config conflict | Check containerd config at `/etc/containerd/config.toml` |
| `nvidia-cuda-validator` | CUDA test failed, driver not functional | Check driver daemonset logs first |
| `gpu-feature-discovery` | Cannot detect GPU properties | Usually follows from a driver issue, fix the driver first |

> **Note:** On k3s with the driver pre-installed on the host, you should not see an `nvidia-driver-daemonset` pod at all (it is skipped when `driver.enabled=false`). On Tanzu, this pod is expected and must reach Running status before other GPU Operator pods can succeed.

**CUDA/driver version mismatch:**

If the GPU test pod (from the Verification section above) fails with CUDA errors, the installed driver version may be outside the compatible range for your Graphistry CUDA image. See the driver compatibility table in the Verification section above.

Check the installed driver version via the GPU Operator's device plugin pod (works on all platforms):

```bash

    -- nvidia-smi --query-gpu=driver_version --format=csv,noheader
```

```

```

**Node taints blocking GPU Operator DaemonSets:**

Some Kubernetes distributions add `NoSchedule` taints to GPU nodes. The GPU Operator tolerates common taints, but custom taints may block its DaemonSets.

```bash
kubectl describe node <node-name> | grep -A5 Taints
```

If you see custom taints, either remove them or add matching tolerations to the GPU Operator helm install.

**Tanzu: VM not provisioned with GPU:**

On Tanzu, if none of the GPU Operator pods are detecting a GPU, the worker node VM may not have been provisioned with a GPU-enabled VM class. Verify at the vSphere level:

1. Check that a VM class with PCI device (passthrough or vGPU) exists
2. Check that the VM class is associated with the vSphere Namespace
3. Check that the TKC worker node pool references the GPU-enabled VM class
4. For passthrough: verify PCI passthrough is toggled on the ESXi host for the GPU device
5. For vGPU: verify the NVIDIA vGPU Manager VIB is installed on the ESXi host

These are vSphere administrator tasks. See the [Tanzu README](tanzu/README.md) for details.

**What are Kubernetes Operators and why does Graphistry need them?**

A Kubernetes [Operator](https://kubernetes.io/docs/concepts/extend-kubernetes/operator/) is a controller that extends the Kubernetes API with Custom Resource Definitions (CRDs). Instead of manually creating pods, services, secrets, and volumes, you describe the desired state in a Custom Resource (CR) and the operator continuously reconciles the cluster to match that state, handling provisioning, failover, backups, and upgrades automatically.

#### Graphistry's Postgres Architecture and the Postgres Operator

PostgreSQL is the central data store for all Graphistry persistent state. Two services connect directly to the same PostgreSQL database with independent connections:

```
                    ┌───────────────────┐
                    │    PostgreSQL     │
                    │ (managed by PGO)  │
                    └─────────┬─────────┘
                              │
                ┌─────────────┴─────────────┐
                │                           │
    ┌───────────┴───────────┐   ┌───────────┴───────────┐
    │        nexus          │   │  streamgl-sessions    │
    │   (Django ORM)        │   │   (Sequelize ORM)     │
    │   Schema OWNER        │   │   Schema CONSUMER     │
    │   runs migrations     │   │   reads/writes same   │
    │   on every startup    │   │   tables directly     │
    └───────────┬───────────┘   └───────────┬───────────┘
                │                           │
     auth gateway, user/org/        session metadata,
     dataset/file management,       visualization state
     internal APIs for other        (layout, filters,
     services, SSO/OAuth2           encodings, snapshots)
```

- **nexus** (Django/Python, port 8000): The central API server and authentication gateway. It **owns the database schema** by running `python manage.py migrate` on every startup, creating and updating all tables. It manages: users, organizations, teams, memberships, datasets, files, sessions, named endpoints (GFQL/UDFs), share links, permissions (Casbin RBAC), SSO/OAuth2/JWT providers, Stripe billing, feature flags (Waffle), and API personal keys. Other services depend on nexus because (1) nginx uses nexus for auth subrequests to validate JWT tokens before proxying to any service, (2) forge-etl-python calls nexus's internal API (`http://nexus:8000/api/v1/internal/`) to register datasets and revisions, and (3) all JWT token verification flows through nexus.

- **streamgl-sessions** (Node.js/Sequelize, port 8080): Manages visualization session lifecycle and state. It connects directly to PostgreSQL (not through nexus) and reads/writes the same `viz_session`, `viz_sessionsnapshot`, `viz_sessionsnapshotstate`, `viz_dataset`, and `viz_datasetrevision` tables that nexus also operates on. It stores the complete visualization state per session: layout parameters, filters, encodings, colors, branding, histograms, and timebars. Other services (streamgl-viz, streamgl-gpu) call streamgl-sessions via HTTP to create, clone, and load sessions. It does NOT own the schema, so its Sequelize models must stay compatible with the Django-managed schema (the source code has `// Must match schema controlled by Nexus` comments in every model file).

Both services (along with every other Graphistry pod) get PostgreSQL credentials from the PGO-managed secret `postgres-pguser-graphistry` via `secretKeyRef` env vars. The secret name is constructed from `{global.postgres.host}-pguser-{global.postgres.user}` (defaults: `postgres` and `graphistry`). See [Section 8a](#8a-nexus-and-postgres-connectivity) for the full env var to secret key mapping.

**Why is PostgreSQL so critical?** If postgres is down, nexus cannot start (it runs migrations on boot), which blocks the entire service startup chain (see Section 8). streamgl-sessions also cannot function without postgres, breaking session creation for all visualizations. This is why the troubleshooting guide starts with verifying postgres health before anything else in Section 7.

**[PGO -- Crunchy Postgres Operator](https://access.crunchydata.com/documentation/postgres-operator/latest/)** ([GitHub](https://github.com/CrunchyData/postgres-operator)) -- **Required**

PGO runs in the `postgres-operator` namespace and watches for `PostgresCluster` CRs across the cluster. When Graphistry's `postgres-cluster` chart is installed, it creates a `PostgresCluster` CR in the `graphistry` namespace. PGO then automatically provisions:

- **Pods**: `postgres-instance1-*` (database with [Patroni](https://github.com/patroni/patroni) for high availability) and `postgres-repo-host-0` (dedicated [pgBackRest](https://pgbackrest.org/) backup repository server)
- **Services**: `postgres-ha` (general cluster access, port 5432), `postgres-primary` (routes to current primary), `postgres-replicas` (routes to read replicas), `postgres-ha-config` (Patroni internal), `postgres-pods` (pod discovery)
- **Secrets**: `postgres-pguser-graphistry` containing connection credentials (host, port, dbname, user, password, connection URI)
- **PVCs**: data volume per instance and a pgBackRest repo volume for backups
- **CronJobs**: automated full, differential, and incremental backups via pgBackRest

[Patroni](https://github.com/patroni/patroni) is an open source PostgreSQL high availability manager created by Zalando. It runs as a sidecar inside each postgres pod (bundled in PGO's database container) and handles automatic leader election, failover, and replica management. Patroni uses a [distributed configuration store (DCS)](https://patroni.readthedocs.io/en/latest/kubernetes.html) to track which instance is the primary. In Kubernetes, the DCS is implemented using Endpoints and ConfigMap annotations, so no external service (etcd, Consul, ZooKeeper) is required. If the primary pod crashes or becomes unresponsive, Patroni automatically promotes a replica to primary and updates the Kubernetes Endpoints so that the `postgres-ha` service routes to the new leader. This failover works even if the PGO operator pod itself is down, because Patroni runs inside the postgres pods.

Without PGO installed first, the `PostgresCluster` CR has no controller to act on it and no postgres pods will be created.

#### Graphistry's Dask Architecture and the Dask Operator

Graphistry uses [Dask](https://www.dask.org/) for GPU-accelerated distributed computing ([cuDF](https://docs.rapids.ai/api/cudf/stable/) DataFrames, Dask SQL queries, GPU ETL pipelines). Three pods are involved, and they all run the **same Docker image** (`graphistry/etl-server-python`) with different entrypoints:

```
forge-etl-python (Graphistry ETL web service)
    │
    │  dask.distributed.Client(address='dask-scheduler:8786')
    │  (plain TCP connection)
    ▼
dask-scheduler (coordinates task distribution, port 8786)
    │
    │  Dask internal protocol (task graph distribution)
    ▼
dask-cuda-worker (executes GPU compute via cuDF/RAPIDS)
    │
    ▼
  GPU(s)
```

- **`forge-etl-python`** (aka `fep` or `etl-server-python`): Graphistry's ETL web service (Quart/Hypercorn). Receives data upload and visualization requests from users, then submits GPU compute jobs to the Dask cluster. It connects to the scheduler via a standard `dask.distributed.Client` TCP connection to `dask-scheduler:8786` (configurable via `DASK_CUDF_CLIENT_ADDRESS` env var). It uses `client.compute()` for [cuDF](https://docs.rapids.ai/api/cudf/stable/) DataFrame operations, `client.submit()` for file parsing, and `client.run()` to initialize GPU memory ([RMM -- RAPIDS Memory Manager](https://docs.rapids.ai/api/rmm/stable/)) on workers at startup. forge-etl-python does not know or care whether the scheduler was created by the Dask Operator or by a plain Kubernetes Deployment.

- **`dask-scheduler`**: Runs the `dask-scheduler` CLI command. Maintains a task graph and distributes work to connected workers. Listens on port 8786 (scheduler protocol) and 8787 (dashboard). It is a coordinator only, it performs no GPU computation itself.

- **`dask-cuda-worker`**: Runs the `dask cuda worker` CLI command via supervisord. Each worker process is pinned to a specific GPU. Executes the actual GPU compute tasks (cuDF transforms, Dask SQL, [RAPIDS](https://rapids.ai/) operations). The number of worker **pods** is controlled by `dask.workers` in your values file (default: 1). Within each pod, the number of worker **processes** is controlled by `DASK_NUM_WORKERS` env var (default: 1), with supervisord managing multi-process round-robin across GPUs.

**Why is the [Dask Kubernetes Operator](https://kubernetes.dask.org/en/stable/)** ([GitHub](https://github.com/dask/dask-kubernetes)) **optional?**

The Graphistry helm chart supports two deployment modes controlled by `dask.operator` in your values file (default: `false`):

| Mode | `dask.operator` | How scheduler/workers are created | Dask Operator required? |
|---|---|---|---|
| Standard (default) | `false` | Plain Kubernetes Deployments and Services | No |
| Operator | `true` | A `DaskCluster` Custom Resource | Yes, must be pre-installed in `dask-operator` namespace |

In **standard mode** (`dask.operator: false`), the helm chart creates plain Kubernetes Deployments for `dask-scheduler` (1 replica) and `dask-cuda-worker` (`dask.workers` replicas), plus their Services. No operator is needed.

In **operator mode** (`dask.operator: true`), the helm chart creates a `DaskCluster` CR instead. The Dask Operator (running in the `dask-operator` namespace) watches for this CR and creates the scheduler pod, scheduler service, and a `DaskWorkerGroup` with the requested number of worker replicas. The operator provides four CRDs:

| CRD | Purpose |
|---|---|
| `DaskCluster` | Primary resource. Creates scheduler + default worker group. |
| `DaskWorkerGroup` | Manages a group of worker pods. Supports `kubectl scale`. Additional groups can be added for heterogeneous workloads (e.g., GPU vs CPU workers). |
| `DaskJob` | Batch jobs. Creates an ephemeral DaskCluster that auto-deletes when the job completes. |
| `DaskAutoscaler` | Monitors the scheduler task queue and auto-scales workers between configurable min/max. |

Both modes produce the same result from forge-etl-python's perspective: a scheduler at `dask-scheduler:8786` with connected GPU workers.

**When should you use operator mode?** Operator mode is useful when you want to use `DaskAutoscaler` for automatic worker scaling based on task queue depth, `DaskJob` for batch workloads, or multiple `DaskWorkerGroup` resources for heterogeneous worker pools. For standard single-node or fixed-size deployments, the default mode is simpler and requires fewer pre-installed components.

**Worker scaling** is manual in the default mode. Set `dask.workers` in your values file to control the number of dask-cuda-worker pods (default: 1). In operator mode, you can additionally create a `DaskAutoscaler` CR for automatic scaling.

## 6. StorageClass

Graphistry requires a StorageClass with `reclaimPolicy: Retain` so data (postgres databases, uploaded files, notebooks, visualizations) is preserved across redeployments. This section covers **single-node** deployments. For multi-node cluster mode, see the [Cluster README](cluster/README.md), which requires `ReadWriteMany` (RWX) volumes and a different StorageClass (default: `retain-sc-cluster`).

**Single-node vs cluster mode storage:**

| Mode | Default StorageClass | Access Mode | Typical Provisioner |
|---|---|---|---|
| Single-node (default) | `retain-sc` | ReadWriteOnce (RWO) | Platform-specific (see table below) |
| Cluster mode (`ENABLE_CLUSTER_MODE: true`) | `retain-sc-cluster` | ReadWriteMany (RWX) | NFS-based (e.g., Longhorn, NFS Ganesha) |

Both modes can be overridden with `global.storageClassNameOverride` in your values file.

**StorageClass requirements:**

| Property | Value | Description |
|---|---|---|
| `reclaimPolicy` | Retain | Data preserved when PVC deleted (manual cleanup required) |
| `volumeBindingMode` | WaitForFirstConsumer | PV created only when a pod needs it |

### Troubleshoot and Debug

**StorageClass creation fails ("already exists"):**

```bash
# Check what already exists
kubectl get sc
kubectl describe sc retain-sc
```

If `retain-sc` exists from a previous deployment, verify it has the correct properties (`Retain`, `WaitForFirstConsumer`, `AllowVolumeExpansion: True`). If not, delete and re-create it:

```bash
kubectl delete sc retain-sc
# Then re-apply the StorageClass manifest from your platform README
```

**StorageClass has reclaimPolicy: Delete instead of Retain:**

This is a critical misconfiguration. Data will be lost when the Helm release is uninstalled. Verify:

```bash
kubectl get sc -o custom-columns=NAME:.metadata.name,RECLAIM:.reclaimPolicy
```

If `retain-sc` shows `Delete`, it was created incorrectly. Delete and re-create it with `reclaimPolicy: Retain`. Note that `reclaimPolicy` cannot be changed on an existing StorageClass, you must delete and re-create it.

**Wrong provisioner for your platform:**

The StorageClass provisioner and the `global.provisioner` helm value must both match your platform's CSI driver. If `kubectl describe sc retain-sc` shows the wrong provisioner, delete and re-create the StorageClass. If the helm value is wrong (default is `kubernetes.io/aws-ebs`), override it in your values file:

```yaml
global:
  provisioner: rancher.io/local-path          # k3s
  # provisioner: pd.csi.storage.gke.io        # GKE
  # provisioner: csi.vsphere.vmware.com       # Tanzu / vSphere
```

Pass the same value to both the `graphistry-helm` and `postgres-cluster` helm installs.

**How StorageClass problems manifest later:**

No pods or PVCs exist at this stage. StorageClass issues only surface later when the postgres-cluster ([Section 7](#7-postgres-cluster)) or Graphistry ([Section 8](#8-graphistry-services)) charts are installed and pods show `Pending` with "no persistent volumes available for this claim". If that happens, come back here to verify the StorageClass is correctly configured.

**StorageClass mismatch between graphistry-helm and postgres-cluster:**

Both charts must use the same StorageClass. If you use `global.storageClassNameOverride` in one chart, pass the same override to both:

```bash
# graphistry-helm
helm upgrade -i g-chart ./charts/graphistry-helm \
    --set global.storageClassNameOverride=your-sc-name \
    --values ./your-values.yaml --namespace graphistry

# postgres-cluster
helm upgrade -i postgres-cluster ./charts/postgres-cluster \
    --set global.storageClassNameOverride=your-sc-name \
    --namespace graphistry
```

**Service roles explained:**

| Service | Type | Purpose |
|---|---|---|
| `postgres-primary` | Headless (ClusterIP: None) | Client-facing service. PGO creates its Endpoints to point to `postgres-ha`'s ClusterIP. DNS resolves `postgres-primary.graphistry.svc` to that ClusterIP. This is the host that nexus and streamgl-sessions use (via the PGO secret's `host` field) |

**How `postgres-primary` routes to the leader pod (two-level indirection):**

PGO uses a two-level routing design to decouple the client-facing service name from Patroni's internal naming constraints. The code is in the PGO source at `internal/controller/postgrescluster/cluster.go`:

```
Client (nexus, streamgl-sessions)
  │
  │ resolves postgres-primary.graphistry.svc via DNS
  ▼
postgres-primary (headless, ClusterIP: None, no pod selector)
  │
  │ PGO-managed Endpoints point to postgres-ha's ClusterIP
  │ (cluster.go line 172: endpoints.Subsets = {{IP: leader.Spec.ClusterIP}})
  ▼
postgres-ha (ClusterIP: 10.43.x.x, no pod selector)
  │
  │ Patroni-managed Endpoints point to the elected leader pod IP
  │ (patroni.go line 239: "let Patroni manage the Endpoints")
  ▼
postgres-instance1-xxxx-0 (leader pod)
```

Why two levels? Patroni requires that its leader lease Endpoints be named after the "scope" (`postgres-ha`). PGO wants a stable, descriptive service name (`postgres-primary`) for clients. Since Patroni controls the `postgres-ha` Endpoints, PGO creates a separate headless service (`postgres-primary`) whose Endpoints redirect to `postgres-ha`'s ClusterIP. This is documented in the source: "To stay free from those constraints, our primary Service resolves to the ClusterIP of the Service created in reconcilePatroniLeaderLease."

The traffic path when nexus connects to `postgres-primary.graphistry.svc`:

```
nexus resolves postgres-primary.graphistry.svc
    -> DNS returns 10.43.x.x (postgres-ha's ClusterIP, from PGO-managed Endpoints)
        -> kube-proxy routes 10.43.x.x to 10.42.x.x (the leader pod, from Patroni-managed Endpoints)
            -> postgres-instance1-xxxx-0 receives the connection
```

Two hops, but the client only sees one connection. During a failover, Patroni updates `postgres-ha`'s Endpoints to point to the new leader pod. Because `postgres-primary` resolves to `postgres-ha`'s ClusterIP (which does not change), clients are automatically routed to the new leader without any DNS change.

> **Warning: Schedule collision.** pgBackRest only allows one backup at a time per stanza. If two backup CronJobs fire at the same minute, the second one fails with `unable to acquire lock on file '/tmp/pgbackrest/db-backup.lock'`. The failed pod stays in `Error` status. While this does not block Graphistry startup (init containers use `pg_isready` which is immune to backup pod status), it means backups are failing silently.
>
> The default schedules are designed to avoid collisions:
>
> | CronJob | Schedule | Fires at |
> |---------|----------|----------|
> | `postgres-repo1-full` | `0 1 * * 0` | Sunday 1:00 AM |
> | `postgres-repo1-diff` | `0 3 * * *` | Daily 3:00 AM |
> | `postgres-repo1-incr` | `15,45 * * * *` | Every hour at :15 and :45 |
>
> The incremental schedule uses `15,45` (not `*/30` which fires at :00 and :30) specifically to avoid overlapping with full (:00) and differential (:00). If you customize backup schedules in your postgres-cluster values, make sure no two schedules can fire at the same minute. Common mistake: using `*/30 * * * *` for incremental, which collides with the differential at 3:00 AM daily and the full at 1:00 AM on Sundays.

The `-c database` flag is required because the postgres pod has 4 containers. Without it, kubectl picks the first container which may not be the database. The `-h 127.0.0.1` flag connects via TCP (not a local socket) because the postgres-cluster chart's pg_hba config only has `host` (TCP) trust rules, not `local` (socket) rules.

> **Security note:** The default pg_hba rule `host all all 0.0.0.0/0 trust` means any pod in the Kubernetes cluster can connect to postgres on port 5432 without a password. This is acceptable when postgres is only reachable within the cluster network (ClusterIP services, no NodePort or LoadBalancer). For production environments that require stricter access control, override the pg_hba rules in the `PostgresCluster` CR to use `scram-sha-256` authentication instead of `trust`.

The `graphistry` database and the `graphistry` user are created by PGO from the `PostgresCluster` CR (defined in the postgres-cluster chart via `global.postgres.name` and `global.postgres.user`). At this point, the database is **empty** (no tables). Tables are created later when nexus runs Django migrations on its first startup ([Section 8a](#8a-nexus-and-postgres-connectivity)). You can verify the database is empty:

```bash
kubectl exec -n graphistry -it $(kubectl get pods -n graphistry \
    -l postgres-operator.crunchydata.com/role=master -o name | head -1) \
    -c database -- psql -h 127.0.0.1 -U graphistry -d graphistry -c "\dt"
```

Expected output (empty table list):

```
Did not find any relations.
```

```bash

```

### Troubleshoot and Debug

**Postgres pod stuck Pending:**

```bash
kubectl describe -n graphistry $(kubectl get pods -n graphistry -o name | grep postgres-instance | head -1)
```

Common causes:
- PGO CRD not installed (check Section 4)
- StorageClass not available (check Section 6)
- Insufficient CPU or memory. PGO reserves resources with Guaranteed QoS. On small nodes (e.g., GKE `n1-highmem-4`), the backup CronJob pods may not fit.

**Postgres pod CrashLoopBackOff:**

```bash
kubectl logs -n graphistry $(kubectl get pods -n graphistry -o name | grep postgres-instance | head -1) -c database
```

Common causes:
- Data directory permissions: PGO expects specific ownership on `/pgdata`. If the PVC was previously used by a different PostgreSQL installation or PGO version, the permissions may not match. Check logs for `FATAL: data directory has wrong ownership` or `initdb: could not change permissions`.
- PVC from a different cluster: If you reinstalled with a different cluster name but reused the same PV, the existing data directory won't match. Either clear the PV or use the same `PostgresCluster` name.
- Insufficient shared memory: Check logs for `could not resize shared memory segment`. This can happen if the container's `shm-size` is too small for the configured `shared_buffers`.

**Postgres pod stuck in Init containers (Init:0/2):**

```bash
# Check which init container is stuck
kubectl describe -n graphistry $(kubectl get pods -n graphistry -o name | grep postgres-instance | head -1) | grep -A5 "Init Containers"

# Check init container logs
kubectl logs -n graphistry $(kubectl get pods -n graphistry -o name | grep postgres-instance | head -1) -c postgres-startup
```

The postgres pod has 2 init containers (`postgres-startup` and `nss-wrapper-init`). If `postgres-startup` is stuck, it is usually a PVC mount issue. If `nss-wrapper-init` is stuck, check for image pull errors.

**Secret `postgres-pguser-graphistry` not appearing:**

If the secret is missing after the postgres pod is running, PGO has not finished reconciling. Check the PGO operator logs:

```bash
kubectl logs -n postgres-operator deployment/pgo -f
```

Common causes:
- PGO operator is not running (check [Section 4](#4-kubernetes-operators-pgo-and-dask))
- The `PostgresCluster` CR has a validation error. Check: `kubectl get postgresclusters -n graphistry -o yaml | grep -A5 status`
- RBAC: PGO needs permission to create secrets in the `graphistry` namespace

**Patroni not initializing or showing bad state:**

If `patronictl list` shows `State: start failed` or `State: stopped`:

```bash
# Check Patroni logs (Patroni logs go to the database container)
# Note: most Patroni messages do not contain the word "patroni", so grep for
# common Patroni keywords like INFO/WARNING/ERROR, "leader", "bootstrap", "failed"
kubectl logs -n graphistry $(kubectl get pods -n graphistry -o name | grep postgres-instance | head -1) \
    -c database | grep -E "INFO:|WARNING:|ERROR:|leader|bootstrap|failed|refusing|timeline"

# Check the DCS Endpoints for the cluster state (Patroni config stored in annotations)
kubectl get endpoints postgres-ha-config -n graphistry -o yaml
```

Common causes:
- Timeline mismatch: if the PVC has data from a previous cluster with a different timeline, Patroni refuses to start. Check logs for `refusing to start` or `timeline`.
- DCS stale: the `postgres-ha-config` Endpoints contain stale annotations from a previous install. Delete and let Patroni recreate: `kubectl delete endpoints postgres-ha-config -n graphistry`

**pg_hba authentication error when connecting via psql:**

The postgres-cluster chart's pg_hba config uses `host all all 0.0.0.0/0 trust` for TCP connections, meaning TCP connections do not require a password. However, there are no `local` (socket) rules for the `graphistry` user, so local socket connections will fail with "no pg_hba.conf entry". Always use `-h 127.0.0.1` (TCP) when connecting via psql:

```bash
# This WORKS (TCP via localhost, covered by "host all all 0.0.0.0/0 trust"):
kubectl exec -n graphistry -it $(kubectl get pods -n graphistry \
    -l postgres-operator.crunchydata.com/role=master -o name | head -1) \
    -c database -- psql -h 127.0.0.1 -U graphistry -d graphistry -c "SELECT 1;"

# This FAILS (local socket, no pg_hba rule for the graphistry user):
kubectl exec -n graphistry -it $(kubectl get pods -n graphistry \
    -l postgres-operator.crunchydata.com/role=master -o name | head -1) \
    -c database -- psql -h /tmp/postgres -U graphistry -d graphistry -c "SELECT 1;"
```

**Backup pod stuck Pending (pgbackrest):**

PGO schedules backup CronJobs. On nodes with limited resources, these pods may not fit.

```bash
kubectl get pods -n graphistry | grep backup
kubectl describe -n graphistry $(kubectl get pods -n graphistry -o name | grep backup | head -1)
```

**Backup pod showing Error or Failed (instead of Completed):**

```bash
# Check backup pod logs
kubectl logs -n graphistry $(kubectl get pods -n graphistry -o name | grep backup | head -1)

# Check the repo-host pod logs (pgBackRest server)
kubectl logs -n graphistry postgres-repo-host-0 -c pgbackrest
```

Common causes:

- **Lock conflict (schedule collision):** Two backups triggered at the same minute (e.g., a differential and an incremental both at :00). The second one fails with:
  ```
  ERROR: [050]: unable to acquire lock on file '/tmp/pgbackrest/db-backup.lock': Resource temporarily unavailable
  HINT: is another pgBackRest process running?
  ```
  pgBackRest only allows one backup per stanza at a time. The data is safe (the other backup completed), and the failed Error pod does NOT block Graphistry init containers because they use `pg_isready` against the `postgres-primary` service, which is immune to backup pod status (see [Key insight 4](#service-startup-order)). To prevent this permanently, fix the CronJob schedules so they never overlap. The default schedules use `15,45` for incremental to avoid :00 collisions. If you see `*/30` in your incremental schedule, update your postgres-cluster values (see the [CronJob schedule warning](#verification-5) above). Check which schedules are active:
  ```bash
  kubectl get cronjobs -n graphistry
  ```

- **Repo volume full:** The 50Gi backup volume may fill up if full backups are large. Check: `kubectl exec -n graphistry postgres-repo-host-0 -c pgbackrest -- df -h /pgbackrest`

- **Repo host not running:** The backup job needs `postgres-repo-host-0` to be Running. Check its status.

> **Important:** Failed and Completed backup pods accumulate the `app=postgres` label, which can block Graphistry service startup. After investigating a failed backup, clean up finished pods:
> ```bash
> kubectl delete pods -n graphistry -l app=postgres --field-selector=status.phase==Succeeded
> kubectl delete pods -n graphistry -l app=postgres --field-selector=status.phase==Failed
> ```

> **Note (Tanzu/vSphere):** On Tanzu, postgres PVC provisioning uses the vSphere CSI driver. If postgres pods are stuck Pending with volume errors, verify the StorageClass provisioner matches your vSphere CSI installation (`csi.vsphere.vmware.com`) and that the StorageClass has `reclaimPolicy: Retain`. The default postgres data volume is 10Gi and the backup volume is 50Gi. These sizes should be sufficient for most deployments, but can be adjusted in your postgres-cluster values file via `postgresCluster.instance1.dataVolumeClaimSpec.resources.requests.storage` and `postgresCluster.backups.pgbackrest.repos.repo1.volume.volumeClaimSpec.resources.requests.storage`. Also note that the default postgres resource limits (100m CPU, 128Mi memory) are intentionally small to allow the chart to run in minimal environments. For production Tanzu deployments, increase these via `global.PostgresResources` in your postgres-cluster values file.


## 8. Graphistry Services

### Service Startup Order

Graphistry services have two types of startup dependencies:

1. **Secret dependency (implicit):** Nearly every Graphistry pod references the PGO secret `postgres-pguser-graphistry` via non-optional `secretKeyRef` env vars. If this secret does not exist (e.g., postgres-cluster chart not installed, or PGO operator not running), pods are stuck in `CreateContainerConfigError` and will never start. This applies to ALL pods except graph-app-kit.

2. **Init container dependencies (explicit):** Init containers that wait for specific pods or services to be ready before the main container starts.

The full dependency chain:

```
PGO secret "postgres-pguser-graphistry" must exist
(created by PGO operator when postgres-cluster chart is installed)
  │
  │  ALL pods below are blocked in CreateContainerConfigError without this secret
  │
  ├── postgres pod (no init container deps, starts first)
  │     │
  │     ├── nexus (init: pg_isready against postgres-primary service)
  │     │     │
  │     │     ├── dask-scheduler (init: waits for nexus service)
  │     │     │     │
  │     │     │     └── forge-etl-python (init: waits for dask-scheduler service + pg_isready)
  │     │     │
  │     │     ├── dask-cuda-worker (init: waits for nexus service)
  │     │     ├── streamgl-gpu (init: waits for nexus service)
  │     │     └── pivot (init: waits for nexus service + pg_isready)
  │     │
  │     └── streamgl-viz (init: pg_isready + redis service wait)
  │
  ├── redis (no init container deps, but blocked without PGO secret)
  ├── caddy (no init container deps, but blocked without PGO secret)
  ├── streamgl-sessions (no init container deps, but blocked without PGO secret)
  ├── notebook (init: volume chown only, no service wait, but blocked without PGO secret)
  └── graph-app-kit-* (init: copy files only, does NOT need PGO secret, uses gak-secret which is optional)

nginx (init: waits for forge-etl-python + streamgl-gpu + streamgl-sessions -- LAST in chain)
```

**Key insight 1:** If the PGO secret does not exist, NOTHING starts (except graph-app-kit). This is the most common cause of a deployment where every pod shows `CreateContainerConfigError`. The fix: install the postgres-cluster chart and wait for PGO to create the secret (`kubectl get secret postgres-pguser-graphistry -n graphistry`).

**Key insight 2:** Even after the secret exists, if the postgres pod is not running, nexus cannot start, which blocks dask-scheduler, dask-cuda-worker, streamgl-gpu, and pivot. If nexus is not running, nginx will never start because forge-etl-python and streamgl-gpu are blocked upstream.

**Key insight 3:** redis, caddy, and streamgl-sessions have no init container deps, but they WILL NOT START if the PGO secret is missing. This is the most confusing failure mode because these pods appear to have "no dependencies" in the init container chain, yet they are stuck. The pod events will show `CreateContainerConfigError: secret "postgres-pguser-graphistry" not found`.

**Key insight 4:** Init containers use `pg_isready` targeting the `postgres-primary` service to check postgres readiness. This approach tests actual PostgreSQL wire-protocol readiness and is immune to backup pod status. The init container logs will show:

```
Waiting for PostgreSQL to be ready...
PostgreSQL not ready, waiting 5 seconds...
PostgreSQL not ready, waiting 5 seconds...
PostgreSQL is up!
```

If the init container is stuck, it means the `postgres-primary` service is not routing to a healthy postgres instance. Check:

```bash
# Is the postgres instance pod running?
kubectl get pods -n graphistry -l postgres-operator.crunchydata.com/data=postgres

# Is the postgres-primary service healthy?
kubectl get endpoints postgres-primary -n graphistry

# Test pg_isready directly from inside the cluster
kubectl run -it --rm pg-test --image=registry.developers.crunchydata.com/crunchydata/crunchy-postgres:ubi8-14.5-1 \
    --restart=Never -n graphistry -- pg_isready -h postgres-primary -p 5432
```

> **Historical note (pre-fix):** Older chart versions used `k8s-wait-for pod -lapp=postgres` which matched ALL pods with that label, including Completed/Failed backup job pods. Since those pods have `Ready=False`, they blocked the wait indefinitely. If you are running an older chart version and see init containers stuck with `Waiting for pod -lapp=postgres...`, upgrade the chart or clean up finished backup pods:
> ```bash
> kubectl delete pods -n graphistry -l app=postgres --field-selector=status.phase==Succeeded
> kubectl delete pods -n graphistry -l app=postgres --field-selector=status.phase==Failed
> ```

### 8a. Nexus and Postgres Connectivity

Nexus is the central API server. If nexus cannot reach postgres, everything downstream is blocked.

**How Graphistry services get postgres credentials:**

Every Graphistry pod (nexus, pivot, streamgl-gpu, forge-etl-python, dask-scheduler, dask-cuda-worker, streamgl-sessions, streamgl-viz, caddy, nginx, redis, notebook) gets its postgres connection details from the PGO-managed secret `postgres-pguser-graphistry` (see the full key list in [Section 7](#7-postgres-cluster)). The secret name is constructed from two Helm values: `{global.postgres.host}-pguser-{global.postgres.user}` (defaults: `postgres` and `graphistry`).

The env vars injected into each pod from this secret:

| Env Var             | Secret Key | Example Value                              |
|---------------------|------------|--------------------------------------------|
| `POSTGRES_HOST`     | `host`     | `postgres-primary.graphistry.svc`          |
| `POSTGRES_PORT`     | `port`     | `5432`                                     |
| `POSTGRES_DB`       | `dbname`   | `graphistry`                               |
| `POSTGRES_USER`     | `user`     | `graphistry`                               |
| `POSTGRES_PASSWORD` | `password` | (auto-generated by PGO)                    |
| `DATABASE_URL`      | `uri`      | `postgresql://graphistry:...@postgres-primary.graphistry.svc:5432/graphistry` |

Notice that `POSTGRES_HOST` resolves to `postgres-primary.graphistry.svc` (from the PGO secret), NOT from the Helm value `global.POSTGRES_HOST`. This means the traffic path is: Graphistry pod -> `postgres-primary` (headless) -> `postgres-ha` (ClusterIP) -> leader pod (as documented in [Section 7](#7-postgres-cluster)).

**Follower mode exception:** In cluster deployments with `global.IS_FOLLOWER: true`, the `POSTGRES_HOST` env var is overridden to use `global.POSTGRES_HOST` directly (e.g., `postgres-ha.graphistry1.svc.cluster.local`) so the follower's pods can reach the leader's postgres across namespaces. All other credentials still come from the PGO secret.

This means the GPU Operator is healthy, but the NVIDIA container runtime is not being used for Graphistry pods. Check:

```bash

    -o jsonpath='{.spec.runtimeClassName}' && echo

```

Expected output: `runtimeClassName` is empty (Graphistry pods do not specify one), and `nvidia` should appear in the runtimeclass list:

```bash

On k3s, the nvidia runtime should be listed and Graphistry templates do not specify `runtimeClassName`, so they rely on the default runtime being nvidia-capable. If the default runtime is not nvidia (configured via `--default-runtime nvidia` in k3s or `NVIDIA_RUNTIME_SET_AS_DEFAULT=true` in the GPU Operator), either configure it as default or add `runtimeClassName: nvidia` to your values override.

### 8c. Caddy, Nginx, and Traffic Flow

**All pods stuck in `CreateContainerConfigError`:**

### Troubleshoot and Debug

**No external IP on LoadBalancer (stays `<pending>`):**

Your cluster does not have a load balancer provisioner. Options:
- k3s: Uses built-in ServiceLB (Klipper) by default
- GKE: Automatically provisions a Google Cloud Load Balancer
- Tanzu/on-prem: Requires NSX Advanced Load Balancer (Avi) or MetalLB

Workaround with NodePort (GKE/Tanzu with nginx-ingress only):

```bash
# Only works if you have nginx-ingress installed (GKE/Tanzu)
kubectl patch svc -n ingress-nginx ingress-nginx-controller \
    -p '{"spec": {"type": "NodePort"}}'
```

On k3s, Traefik's ServiceLB (Klipper) handles external access automatically. There is no `ingress-nginx` namespace on k3s.

Workaround with port-forward (quick local access, any platform):

```bash
kubectl port-forward -n graphistry svc/caddy-graphistry 8080:80
# Then open http://localhost:8080 in your browser
```

**502/504 errors:**

Backend pods are not ready. Check nexus and nginx status:

```bash
kubectl get pods -n graphistry | grep -E "nexus|nginx"
kubectl logs -n graphistry $(kubectl get pods -n graphistry -o name | grep nexus) --tail=50
kubectl logs -n graphistry $(kubectl get pods -n graphistry -o name | grep nginx) --tail=50
```

Expected output when healthy:

```
nexus-7d54f79547-js42f   1/1   Running   0   53m
nginx-7496674fc7-cxxgl   1/1   Running   0   100m
```

Healthy nexus logs show a repeating pattern of API requests returning HTTP 200:

```bash
"GET /ht/ HTTP/1.1" 200 2435
"GET /api/v1/csrfmiddlewaretoken/ HTTP/1.1" 200 677
"GET /api/v1/my/user/entitlements/ HTTP/1.1" 200 114
"GET /wafflejson HTTP/1.1" 200 588
"GET / HTTP/1.1" 200 17968
```

The `/ht/` endpoint is the health check (called every 2 seconds by the liveness probe). The `csrfmiddlewaretoken`, `entitlements`, and `wafflejson` requests are periodic frontend polling. `GET /` returning `200 17968` is a successful page load (~17KB HTML).

Healthy nginx logs show liveness probe checks and forwarded requests:

```bash
127.0.0.1 - - [20/Feb/2026:09:53:12 +0000] "GET /healthz HTTP/1.1" 201 0 "-" "curl/7.88.1"
10.42.0.59 - - [20/Feb/2026:09:54:08 +0000] "GET / HTTP/1.1" 200 17968 "-" "curl/8.12.1"
```

The `127.0.0.1` requests are internal liveness probes (HTTP 201 is healthy). The `10.42.0.59` requests are traffic forwarded from Caddy (that IP is the Caddy pod). If you see 502 or 503 responses in the nginx access log, the upstream service (nexus, streamgl-viz, etc.) is not responding.

**Graphistry loads but visualizations fail:**

Check GPU services ([Section 8b](#8b-gpu-services-streamgl-gpu-forge-etl-python-dask-cuda-worker)) and verify streamgl-gpu and forge-etl-python are healthy.


## 10. Telemetry Stack (Optional)

The telemetry stack is deployed when `global.ENABLE_OPEN_TELEMETRY: true` is set in your Helm values file. By default it is disabled.

**Architecture:**

Graphistry services export telemetry data (metrics and traces) using the [OpenTelemetry](https://opentelemetry.io/) standard. All telemetry flows through the OpenTelemetry Collector (`otel-collector`), which dispatches to backend observability tools depending on the deployment mode.

**Local stack mode** (`OTEL_CLOUD_MODE: false`, default):

All 6 observability components are deployed in-cluster:

```
Graphistry Services (nexus, streamgl-viz, streamgl-gpu, forge-etl-python, ...)
        │
        │ (OTLP push)
        ▼
  otel-collector
        │
        ├───> Prometheus (metrics storage and querying)
        ├───> Jaeger (distributed tracing)
        └───> Grafana (visualization, reads from Prometheus)

  dcgm-exporter ────> Prometheus (GPU metrics via scrape)
  node-exporter ────> Prometheus (system metrics via scrape)
```

**Cloud mode** (`OTEL_CLOUD_MODE: true`):

Only the otel-collector is deployed in-cluster. It forwards all telemetry to an external cloud observability platform (Grafana Cloud, Datadog, etc.) via OTLP. Prometheus, Jaeger, Grafana, DCGM Exporter, and Node Exporter are not deployed.

```
Graphistry Services (nexus, streamgl-viz, streamgl-gpu, forge-etl-python, ...)
        │
        │ (OTLP push)
        ▼
  otel-collector
        │
        │ (OTLP/HTTP export)
        ▼
  External Cloud Platform (Grafana Cloud, Datadog, New Relic, etc.)
```

**Components:**

| Component | Default Image | Purpose |
|-----------|---------------|---------|
| OpenTelemetry Collector | `otel/opentelemetry-collector-contrib:0.87.0` | Central telemetry collection and forwarding |
| Prometheus | `prom/prometheus:v2.47.2` | Metrics storage and querying |
| Grafana | `grafana/grafana:11.0.0` | Visualization and dashboarding (DCGM GPU dashboard pre-configured) |
| Jaeger | `jaegertracing/all-in-one:1.50.0` | Distributed tracing for the graph rendering flow |
| DCGM Exporter | `nvcr.io/nvidia/k8s/dcgm-exporter:3.3.5-3.4.1-ubuntu22.04` | NVIDIA GPU metrics (temperature, utilization, memory, power) |
| Node Exporter | `prom/node-exporter:v1.8.2` | System-level metrics (CPU, memory, disk, network) |

**Deployment modes:**

| Mode | Helm Value | What Gets Deployed |
|------|------------|--------------------|
| Local stack (default) | `telemetryStack.OTEL_CLOUD_MODE: false` | All 6 components above |
| Cloud forwarding | `telemetryStack.OTEL_CLOUD_MODE: true` | Only otel-collector, forwards to external OTLP endpoint |
| Hybrid | Custom Helm chart adjustments | Local stack + forwarding to external endpoint |

For cloud mode, set these additional values:

```yaml
telemetryStack:
  OTEL_CLOUD_MODE: true
  openTelemetryCollector:
    OTEL_COLLECTOR_OTLP_HTTP_ENDPOINT: "https://hostname.grafana.net/otlp"
    OTEL_COLLECTOR_OTLP_USERNAME: "<instance-id>"
    OTEL_COLLECTOR_OTLP_PASSWORD: "<api-token>"
```

**Cluster mode telemetry:** In multi-cluster deployments (`ENABLE_CLUSTER_MODE: true`), all follower OTEL collectors export to the leader's collector via `telemetryStack.openTelemetryCollector.LEADER_OTEL_EXPORTER_OTLP_ENDPOINT` (e.g., `otel-collector.graphistry1.svc.cluster.local:4317`). The leader's collector then dispatches to Prometheus/Jaeger/Grafana.

**Feature flag:** Telemetry tracing is controlled per-user by the `flag_ot_traces` waffle feature flag in the Graphistry admin panel (`/admin/waffle/flag/`). This flag is **off by default**. Even with `ENABLE_OPEN_TELEMETRY: true`, no traces will be exported until this flag is enabled for the target users. Metrics (Prometheus, DCGM, Node Exporter) are not affected by this flag and are always collected when the stack is enabled.

`worker_read_crashes_total` tracks GPU worker crashes in `streamgl-gpu`. A non-zero and increasing value indicates recurring GPU worker failures. The `forge_etl_python_upload_*` metrics track file uploads and dataset creation activity in the ETL pipeline.

### Troubleshoot and Debug

**DCGM profiling error on GKE or Tanzu vGPU environments:**

This is a known issue where the DCGM profiling module is incompatible with certain environments (vGPU does not support profiling counters). The symptom is `dcgm-exporter` pod CrashLoopBackOff or error logs mentioning `DCGM_ST_NOT_SUPPORTED` or `profiling is not supported`. See the DCGM fix section in the [GKE README](gke/README.md) or [Tanzu README](tanzu/README.md) for the configmap patch workaround that disables profiling metrics.

```bash
kubectl logs -n graphistry $(kubectl get pods -n graphistry -o name | grep dcgm)
```

**No traces appearing in Jaeger:**

1. Check the `flag_ot_traces` waffle feature flag is enabled: log into Graphistry as admin, go to `/admin/waffle/flag/`, and verify `flag_ot_traces` is active for the target users (or set to "Everyone: Yes"). This flag is **off by default**.
2. Trigger a trace by opening a dataset URL in the browser (e.g., `http://<ADDRESS>/graph/graph.html?dataset=<id>`).
3. Check the OTEL Collector logs for export errors:

```bash
kubectl logs -n graphistry $(kubectl get pods -n graphistry -o name | grep otel-collector) | grep -i error
```

4. Verify the OTEL Collector is forwarding to Jaeger:

```bash
kubectl logs -n graphistry $(kubectl get pods -n graphistry -o name | grep otel-collector) | grep -i jaeger
```

**Telemetry pods not starting:**

```bash
kubectl logs -n graphistry $(kubectl get pods -n graphistry -o name | grep otel-collector)
kubectl logs -n graphistry $(kubectl get pods -n graphistry -o name | grep grafana)
kubectl logs -n graphistry $(kubectl get pods -n graphistry -o name | grep prometheus)
kubectl logs -n graphistry $(kubectl get pods -n graphistry -o name | grep jaeger)
kubectl logs -n graphistry $(kubectl get pods -n graphistry -o name | grep dcgm)
kubectl logs -n graphistry $(kubectl get pods -n graphistry -o name | grep node-exporter)
```

Common causes: image pull failures (check `docker-secret-prod`), insufficient resources (telemetry pods add CPU/memory overhead), PVC issues (if persistence is configured).

**Prometheus targets showing status DOWN:**

```bash
kubectl port-forward -n graphistry svc/prometheus 9090:9090
# Then open http://localhost:9090/prometheus/targets in your browser
```

Check that all scrape targets show status `UP`. If a target is DOWN:
- `otel-collector:8889` down: OTEL Collector pod not running or metrics port not exposed
- `dcgm-exporter:9400` down: DCGM Exporter pod not running (check for profiling error above)
- `node-exporter:9100` down: Node Exporter pod not running

**Cloud mode: data not reaching external service (Grafana Cloud, etc.):**

```bash
# Check collector logs for OTLP export errors
kubectl logs -n graphistry $(kubectl get pods -n graphistry -o name | grep otel-collector) \
    | grep -E "error|failed|401|403|timeout" -i

# Verify cloud mode env vars are set
kubectl exec -n graphistry $(kubectl get pods -n graphistry -o name | grep otel-collector) \
    -- printenv | grep OTEL_COLLECTOR
```

Common causes: wrong endpoint URL, expired or invalid API token, network policy blocking outbound HTTPS to the external service. The OTEL Collector logs will show the HTTP status code from the external endpoint.

**Cluster mode: follower telemetry not reaching leader:**

In multi-cluster deployments, verify the follower's OTEL Collector can reach the leader:

```bash
# On the follower cluster
kubectl exec -n graphistry $(kubectl get pods -n graphistry -o name | grep otel-collector) \
    -- wget -qO- http://<leader-otel-collector-endpoint>:13133/ 2>/dev/null

# Check follower collector logs
kubectl logs -n graphistry $(kubectl get pods -n graphistry -o name | grep otel-collector) \
    | grep -E "error|failed|refused" -i
```

The `LEADER_OTEL_EXPORTER_OTLP_ENDPOINT` must be reachable from the follower namespace (e.g., `otel-collector.graphistry1.svc.cluster.local:4317`). If the leader cluster uses network policies, ensure the follower's namespace is allowed to connect on port 4317.

**Grafana shows "No data" on dashboards:**

1. Check the Prometheus datasource is configured: `http://<ADDRESS>/grafana/datasources`
2. Check Prometheus has data: query `up` in the Prometheus UI, it should return entries for all scrape targets
3. If DCGM dashboard shows no data but other dashboards work, the DCGM Exporter is the problem (see DCGM troubleshooting above)


## 11. Volume Binding on Redeployment

When updating, preserve existing volume bindings so that data (postgres, uploads, notebooks, visualizations) persists across redeployments. The StorageClass uses `reclaimPolicy: Retain`, which means PVs are not deleted when PVCs are removed. However, after a `helm uninstall` and reinstall, the new PVCs will not automatically rebind to the existing PVs unless you explicitly set the `volumeName` values.

### Verification

```bash
kubectl get pv
```

Expected output (all Bound, RECLAIM POLICY = Retain):

```bash
NAME                                       CAPACITY   ACCESS MODES   RECLAIM POLICY   STATUS   CLAIM                                       STORAGECLASS   AGE
pvc-4e4f7885-968f-43ee-81ee-d193da4da48f   10Gi       RWO            Retain           Bound    graphistry/postgres-instance1-ngz7-pgdata   retain-sc      6h22m
pvc-6327f91b-8c7f-4644-93b1-e3eb6f79b49a   64Gi       RWO            Retain           Bound    graphistry/data-mount                       retain-sc      108m
pvc-76680b21-4688-4c81-a79e-b01317a6f4db   4Gi        RWO            Retain           Bound    graphistry/local-media-mount                retain-sc      108m
pvc-7e49e1e1-de5f-4847-821b-041d4376d941   4Gi        RWO            Retain           Bound    graphistry/gak-private                      retain-sc      108m
pvc-ac45c3a0-4e57-422c-ba1a-f8a6b2f08754   40Gi       RWO            Retain           Bound    graphistry/uploads-files                    retain-sc      108m
pvc-af477842-6d84-4e26-a6cf-44c52784669e   50Gi       RWO            Retain           Bound    graphistry/postgres-repo1                   retain-sc      6h22m
pvc-fbcee9c7-c985-4c73-8f79-74943eae87a3   4Gi        RWO            Retain           Bound    graphistry/gak-public                       retain-sc      108m
```

All PVs should show `STATUS: Bound` and `RECLAIM POLICY: Retain`. If any PV shows `STATUS: Released`, it means the PVC was deleted but the PV still holds the data (see Troubleshoot section below).

```bash
kubectl get pvc -n graphistry
```

All PVCs should show `STATUS: Bound`.

### Preserving Data Across Redeployments

Before running `helm upgrade` or after a `helm uninstall` + reinstall, generate the `volumeName` block to pin each PVC to its existing PV:

```bash
echo "volumeName:
  dataMount: $(kubectl get pvc data-mount -n graphistry -o jsonpath='{.spec.volumeName}')
  localMediaMount: $(kubectl get pvc local-media-mount -n graphistry -o jsonpath='{.spec.volumeName}')
  uploadsFiles: $(kubectl get pvc uploads-files -n graphistry -o jsonpath='{.spec.volumeName}')
  gakPublic: $(kubectl get pvc gak-public -n graphistry -o jsonpath='{.spec.volumeName}')
  gakPrivate: $(kubectl get pvc gak-private -n graphistry -o jsonpath='{.spec.volumeName}')"
```

Expected output:

```yaml
volumeName:
  dataMount: pvc-6327f91b-8c7f-4644-93b1-e3eb6f79b49a
  localMediaMount: pvc-76680b21-4688-4c81-a79e-b01317a6f4db
  uploadsFiles: pvc-a417fc34-2128-4824-99de-a3267e21e4dc
  gakPublic: pvc-fbcee9c7-c985-4c73-8f79-74943eae87a3
  gakPrivate: pvc-7e49e1e1-de5f-4847-821b-041d4376d941
```

Copy the output into your values file (e.g., `k3s_example_values.yaml`), then run the normal upgrade command:

```bash
helm upgrade -i g-chart ./charts/graphistry-helm \
    --values ./charts/values-overrides/examples/k3s/k3s_example_values.yaml \
    --namespace graphistry --create-namespace
```

This ensures the new PVCs bind to the same PVs, preserving all existing data.

### Troubleshoot and Debug

**PV in Released state after helm uninstall:**

This is normal behavior with `reclaimPolicy: Retain`. The PV preserves your data but will not automatically rebind to a new PVC. Use the `volumeName` approach above to explicitly bind PVCs to the existing PVs on reinstall.

**PVC Pending after reinstall (PV claimRef still points to old PVC):**

If you did not set `volumeName` in your values file and the PVC stays Pending, the PV's `claimRef` still points to the old (deleted) PVC. After `helm uninstall`, PVs take up to 60 seconds to transition from `Bound` to `Released`. Watch for the transition before clearing the claimRef:

```bash
# Watch until PVs show Released (Ctrl+C once you see them)
kubectl get pv --watch | grep Released

# Then clear the stale claimRef
kubectl get pv -o json | \
    jq -r '.items[] | select(.status.phase=="Released") | select(.spec.claimRef.namespace=="graphistry") | .metadata.name' | \
    xargs -I {} kubectl patch pv {} -p '{"spec":{"claimRef": null}}'
```

After patching, the PVCs should automatically bind to the freed PVs. However, using `volumeName` in the values file (as shown above) is the recommended approach because it avoids this issue entirely.

**Stale volumeName pointing to deleted PVs (fresh deployment after full cleanup):**

If you deleted everything (PVs, PVCs, namespace) and then redeployed, but your values file still has `volumeName` entries from the previous deployment, all pods (except postgres and telemetry DaemonSets) will be stuck in `Pending`:

```bash
kubectl get pods -n graphistry
```

Output:
```bash
NAME                              READY   STATUS    AGE
caddy-graphistry-xxx              0/1     Pending   10m
nexus-xxx                         0/1     Pending   10m
dask-scheduler-xxx                0/1     Pending   10m
...
postgres-instance1-xxx            4/4     Running   11m   # PGO manages its own PVCs
dcgm-exporter-xxx                 1/1     Running   10m   # DaemonSets with no PVC
```

The PVCs will show `Pending` with a VOLUME column referencing PVs that no longer exist:

```bash
kubectl get pvc -n graphistry
```

Output:
```bash
NAME             STATUS    VOLUME                                     STORAGECLASS
data-mount       Pending   pvc-6327f91b-8c7f-4644-93b1-e3eb6f79b49a   retain-sc
gak-private      Pending   pvc-7e49e1e1-de5f-4847-821b-041d4376d941   retain-sc
```

And `kubectl describe pod` will show:

```bash
Events:
  Warning  FailedScheduling  default-scheduler  0/1 nodes are available: pod has unbound immediate PersistentVolumeClaims. not found
```

The root cause is `volumeName` in your values file pinning PVCs to PVs that were deleted.

```yaml
# Before (stale values from previous deployment):
volumeName:
  dataMount: pvc-6327f91b-8c7f-4644-93b1-e3eb6f79b49a
  localMediaMount: pvc-76680b21-4688-4c81-a79e-b01317a6f4db
  uploadsFiles: pvc-a417fc34-2128-4824-99de-a3267e21e4dc
  gakPublic: pvc-fbcee9c7-c985-4c73-8f79-74943eae87a3
  gakPrivate: pvc-7e49e1e1-de5f-4847-821b-041d4376d941
```

For a fresh deployment, comment out or remove the `volumeName` block in your values file:
```yaml
# After (for fresh deployment):
#volumeName:
#  dataMount:
#  localMediaMount:
#  uploadsFiles:
#  gakPublic:
#  gakPrivate:
```

Then delete the stuck PVCs and redeploy:
```bash
kubectl delete pvc data-mount local-media-mount gak-public gak-private uploads-files -n graphistry
```

Then redeploy using the standard upgrade command from your platform README (for example, for k3s):
```bash
helm upgrade -i g-chart ./charts/graphistry-helm \
    --values ./charts/values-overrides/examples/k3s/k3s_example_values.yaml \
    --namespace graphistry --create-namespace
```

The StorageClass provisioner will create fresh PVs and the PVCs will bind normally. Once the deployment is healthy, generate new `volumeName` entries (see [Preserving Data Across Redeployments](#preserving-data-across-redeployments) above) for future redeployments.
